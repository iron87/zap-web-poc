

# ZAP Web Playground

This repository provides three test environments (playgrounds) for experimenting with automated web security scans using OWASP ZAP:

- **Form Login**: Flask application with classic form authentication.
- **Local SSO**: Local Single Sign-On authentication using Keycloak and oauth2-proxy.
- **API + OpenAPI**: FastAPI service with Swagger UI and OpenAPI-driven passive scan.

## Features

- Automated ZAP execution via scripts and YAML plans.
- HTML and JSON report generation.
- Authentication testing (classic and SSO).
- API testing via OpenAPI import.
- Template for testing with external Identity Providers (e.g., Microsoft Entra/Office365).

## Requirements

- Docker
- Docker Compose

## Folder Structure

- `app/`: Flask test application
- `api/`: FastAPI test API with Swagger (`/docs`) and OpenAPI (`/openapi.json`)
- `tests/form-login/automation.yaml`: ZAP plan for classic form login
- `tests/sso-local/automation.yaml`: ZAP plan for local SSO
- `tests/sso-local/automation-external-idp-template.yaml`: template for SSO with external IdP
- `tests/api-openapi/automation.yaml`: OpenAPI/passive-scan automation plan reference
- `keycloak/realm-playground.json`: Keycloak configuration with demo client and user
- `scripts/form-login/`: scripts for the form-login playground
- `scripts/sso-local/`: scripts for the SSO playground
- `scripts/api-openapi/`: scripts for the API OpenAPI playground
- `reports/`: reports generated by ZAP scans

## How to Launch the Playgrounds

### Form Login

1. Start the environment:
  ```sh
  ./scripts/form-login/01-up.sh
  ```
2. Run the ZAP scan:
  ```sh
  ./scripts/form-login/02-scan.sh
  ```
3. Stop the environment:
  ```sh
  ./scripts/form-login/03-down.sh
  ```
4. Or run the full sequence:
  ```sh
  ./scripts/form-login/00-all.sh
  ```

## SSO Architecture (Keycloak)

```text
Browser
  |
  | 1) GET /private
  v
sso-public (oauth2-proxy, localhost:4180)
  |
  | 2) OIDC auth redirect
  v
Keycloak (localhost:8090)
  |
  | 3) user login + callback code
  v
sso-public (oauth2 callback)
  |
  | 4) code -> token exchange (against internal keycloak)
  v
Keycloak (service DNS: keycloak:8080)
  |
  | 5) valid session, proxy to upstream
  v
App Flask (app:8080)
```

ZAP path (inside Docker):
```text
ZAP browser auth -> sso (internal service) -> Keycloak -> sso -> App
```

## ZAP files used

Form-login test:
- `tests/form-login/automation.yaml`

Local SSO test:
- `tests/sso-local/automation.yaml`
- called by `scripts/sso-local/02-scan.sh`

API OpenAPI test:
- called by `scripts/api-openapi/02-scan.sh`
- scan command: `zap-api-scan.py -f openapi -t http://api:8081/openapi.json -S`

How ZAP logs in during the SSO test:
1. It uses `authentication.method: browser`.
2. It opens `loginPageUrl: http://sso:4180/private`.
3. It submits credentials `demo-user` / `demo-pass`.
4. It follows OIDC redirects to Keycloak and callback.
5. It runs passive scan first (`passiveScan-wait`) on collected responses.
6. It continues with active scan as an authenticated session.

Job order used in tests:
- `spider -> passiveScan-wait -> activeScan -> report`

API OpenAPI flow used by scripts:
- imports Swagger/OpenAPI from `http://api:8081/openapi.json`
- runs in safe mode (`-S`), so it executes passive scan only
- generates HTML and JSON report files in `reports/`

## What `automation-external-idp-template` is for

`tests/sso-local/automation-external-idp-template.yaml` is not a typo:
- it is an example to copy/adapt when using a real external IdP (not local Keycloak)
- it is not used by default scripts
- it requires replacing URLs, username/password, and verification rules

## Verify API Calls Were Actually Made

The script `scripts/api-openapi/02-scan.sh` already performs a hard check after the scan:
- it reads container logs from service `api`
- it searches for API calls (`GET /openapi.json`, `GET /api/health`, `GET /api/accounts`, `GET /api/customers/...`, `POST /api/transfer`)
- it exits with error if no calls are detected

You can verify manually as well:

```sh
docker compose logs --no-color api | grep -E 'GET /openapi.json|GET /api/health|GET /api/accounts|GET /api/customers/|POST /api/transfer'
```

If that command prints lines, the API endpoints were called and the passive scanner had HTTP messages to process.
